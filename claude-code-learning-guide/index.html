初日

とにかく動かす

まずはインストールして起動し、自分のコードについて話しかけてみるところまで。難しい設定は後回しで大丈夫です。

1. インストール

Node.js が入っていれば、npm でグローバルにインストールできます。

npm install -g @anthropic-ai/claude-code

2. プロジェクトで起動する

作業したいリポジトリのルートフォルダに移動してから claude と打つのがコツ。最初にブラウザでのログインを求められます。

cd ~/projects/my-app claude

3. コードベースについて聞いてみる

いきなりコードを書かせなくてOK。まずは「このリポジトリが何をするものか説明して」と聞いて、Claude がどうやってファイルを読むのかを体験しましょう。

このリポジトリの構成と主要な処理を説明して
💡 初日のコツ

Claude Code はファイルを勝手に書き換える前に必ず確認を求めます。怖がらず、まずは提案内容を眺めるだけでも十分。合わないと思ったら Esc でいつでも止められます。

3日目

基本操作に慣れる

スラッシュコマンドと会話の管理を覚えると、作業がぐっと快適になります。「一度に一つのタスク」を意識しましょう。

1. スラッシュコマンドを覚える

/ から始まる特別なコマンドがあります。まず /help で一覧を確認しましょう。

/help

2. 会話をこまめにリセットする

別の作業に移るときは /clear で会話をまっさらに。前の話題が残っていると回答がぶれる原因になります。

/clear

3. 差分を確認して承認する

ファイル編集が提案されると差分（変更内容）が表示されます。中身を読んでから承認する習慣をつけましょう。意図と違えば、そのまま「ここはこう直して」と追加で指示できます。

💡 3日目のコツ

お願いは小さく区切るほど精度が上がります。「機能A→テスト→機能B」と一つずつ。うまくいかないときは /clear して指示を言い直すのが一番の近道です。

7日目

自分好みに設定する

プロジェクトのルールを覚えさせ、指示の出し方を洗練させる段階。ここを押さえると出力の質が段違いになります。

1. CLAUDE.md を作る

/init を実行すると、プロジェクト用の CLAUDE.md が自動生成されます。ここに書いたルール（コーディング規約・使うコマンド・注意点）は毎回の会話に読み込まれます。

/init

2. 権限（許可）を管理する

毎回同じ確認が出て面倒なら、よく使うコマンドを許可リストに追加できます。設定は /permissions から。安全な範囲で少しずつ広げましょう。

/permissions

3. 「計画→実行」で頼む

大きめの変更は、いきなり書かせず「まず計画を立てて。実装はしないで」と頼むのがコツ。計画を確認してから「ではこの方針で進めて」と続けると、手戻りが激減します。

認証機能を追加したい。まず実装計画だけ立てて。コードは書かないで
💡 7日目のコツ

CLAUDE.md には「守ってほしいこと」を箇条書きで簡潔に。長い散文より短い命令形が効きます。例：「テストは vitest を使う」「新しい依存を入れる前に確認する」。

14日目

ワークフローを広げる

Git 連携や外部ツール接続、自分専用コマンドで、日常の開発に本格的に組み込んでいきます。

1. Git を任せる

「変更をコミットして」と頼めば、差分を読んで適切なコミットメッセージまで作ってくれます。プルリクの下書きも依頼できます。

今の変更を意味のある単位でコミットして

2. MCP で外部ツールをつなぐ

MCP（Model Context Protocol）サーバーを追加すると、DB やチケット管理などの外部システムを Claude から扱えます。設定状況は /mcp で確認。

/mcp

3. 自分専用のコマンドを作る

よく使う指示は .claude/commands/ に Markdown で保存すると、/コマンド名 で呼び出せるようになります。

echo "変更点のテストを書いて実行して" > .claude/commands/test.md
💡 14日目のコツ

スクリーンショットや画像を貼り付けて「この画面と同じUIを作って」「このエラーを直して」と頼めます。バグ報告やデザイン再現で強力です。

30日目

自動化とチーム運用

対話だけでなく、スクリプトや CI に組み込み、チームで使いこなす段階。ここまで来れば立派な Claude Code ユーザーです。

1. スクリプトから呼び出す（ヘッドレス）

-p オプションで、対話画面を開かずに一発で実行できます。シェルスクリプトや CI に組み込めます。

claude -p "READMEを最新のコードに合わせて更新して"

2. モデルを使い分ける

難しい設計は高性能モデル、軽い作業は速いモデル、と /model で切り替えると、速度とコストのバランスが取れます。

/model

3. チームで設定を共有する

CLAUDE.md や .claude/commands/ をリポジトリにコミットすれば、チーム全員が同じルール・同じコマンドで作業できます。オンボーディングも一気に楽に。

💡 30日目のコツ

大きなタスクは「TODOリストを作って一つずつ進めて」と頼むと、進捗が見えて抜け漏れが減ります。長時間の作業でも迷子になりません。
